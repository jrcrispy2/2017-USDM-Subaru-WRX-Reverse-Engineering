I had the chance to read the ECU.

It did respond to SSM on CAN queries as I got a few logging parameters show up, but only 4.

This ECU (22765AG230) appears to be made by Hitachi using a Renesas SH72543 SH-2A processor which has a 2MB ROM and 128kB RAM.
Attachment:
LV9N001D.hex.zip [894.4 KiB]
Downloaded 907 times


I've managed to extract most all of the 2D and 3D tables into a RR Editor definition. These tables are all presented in the ROMs nature units which is integer. The tables have not be verified for correctness meaning the sign of the data and/or data element size could be wrong.
This definition does not contain any data series which is data that would have a dynamically created axis derived during value lookup (the old 16 bit ROMs had many of these called static axis). There are no single element values defined such as enable/disable and thresholds, etc. These values are intermixed with table data or defined in instruction operands (like the old 16 bit ROMs).
Have a look and let me know what you think. RR requires a lot of memory to open this ROM and it will be a bit slow when displaying the tables. Also note that many tables are related to OBD operations and not part of the engine calibration.

The return value from the table lookup is written to 0xFFF8A764 which might be related to WGDC in some way, I haven't followed this code through yet.

For the other two tables the references are:
0xF2B48 references 0xFFF8A2BE. Then there's a call to sub_F3294 where Table_50AA0 is referenced. Then the call to sub_18CCE4.
Table result stored at 0xFFF8A776.

0xF2C42 references 0xFFF8A2BE. Then there's a call to sub_F39E8 where Table_50AF8 is referenced. Then the call to sub_18CCE4.
Table result stored at 0xFFF8A784.

For all three tables Engine Speed is the x-axis.
