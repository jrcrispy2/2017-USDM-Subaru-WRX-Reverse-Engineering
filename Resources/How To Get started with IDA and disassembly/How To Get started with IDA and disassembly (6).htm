I've found a fantastic trick to resolve indirect addressing (well, for those of us who have not yet upgraded to IDA 6.3), using user defined offsets.

for example:

Code:
mov.l   #unk_FFFF636C, r0
ldc     r0, gbr
add     #(unk_FFFF6360 - unk_FFFF636C), r15
mov.l   #Ram_Vehicle_Speed_0, r2
fmov.s  @r2, fr15
mov.l   #Ram_Memorised_Cruise_Speed_, r2
fmov.s  @r2, fr5
fmov    fr5, fr14


Copy the address of the base - GBR, or whatever base is used. Select the indirect reference, and press control-R to get to the user defined offset GUI window. Paste the address into the base address field and enter - it nicely resolves the reference for you, and as a bonus, also creates an X-ref. Winner!

Post subject:  Re: How To: Get started with IDA and disassembly
That's how it turned out for me as well. If there's a way to get 100% disassembly, I don't know what it is. But I would like to.

You can try pressing 'c' to mark non-code sections as code. If it really was code, that usually works well. Other times it will just find one or two opcodes and then give up because the next two bytes are illegal. You can also re-do the analysis pass after you've manually labeled more sections as code. That will usually help it find more code on its own.

I think the issue is that there are chunks of code that are never executed directly from the core code. For example, interrupt routines. The CPU will interrupt whatever is running and jump to those routines based on the interrupt handler vector. It would be handy to have a way to tell IDA "here's an array of pointers to code, please disassemble the code at each address." That's probably something that could be done from an IDC script, but I haven't tried it.
